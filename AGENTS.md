# AGENTS.md

本文件定义 DocVault 仓库中所有 AI 编码代理、自动化代理和人工贡献者必须遵守的全局规则。它是实现约束，不是建议。

## 1. 项目使命

DocVault 是一款面向单用户、多设备、局域网/NAS 环境的跨平台文档自动备份与版本管理工具。

产品由两个独立运行、独立发布的项目组成，并放在同一个 monorepo 中：

1. **桌面客户端**：Tauri + React 桌面界面，以及 Python 本地备份 Agent。
2. **局域网服务端**：FastAPI 服务、PostgreSQL 元数据、NAS 文件存储和预览转换 Worker。

客户端与服务端通过版本化 HTTP API 通信。不要将两者合并成同一个进程、同一个数据库或同一套业务入口。

## 2. 不可破坏的产品不变量

任何实现、重构或依赖升级都不得违反以下规则。

### 2.1 这是备份，不是同步

- 服务端不会把某台设备的文件变化自动下发到其他设备。
- 恢复操作必须由用户显式发起。
- 不实现双向同步、自动合并或冲突解决。
- 不得因为名称中出现“同步”“sync”而引入同步语义。

### 2.2 每台设备的文件树完全独立

文件身份至少由以下信息共同确定：

```text
device_id + backup_root_id + relative_path
```

- 相同相对路径在不同设备上是不同文件。
- 禁止把不同设备的同名路径合并成“逻辑文档”。
- 禁止创建跨设备主线历史或版本分支。
- UI 必须先展示设备，再展示该设备的备份根目录和文件树。

### 2.3 防抖时间固定为 10 分钟

- 文件停止变化 **10 分钟** 后才进入哈希与上传流程。
- 在 10 分钟内再次变化时，重新计时。
- 不得把防抖偷偷改回 5 秒，也不得按每次文件系统事件创建版本。
- 每 **30 分钟**执行一次补偿扫描，用于修复监听事件丢失。
- 防抖状态和待处理任务必须持久化到客户端 SQLite，重启后能够继续。

### 2.4 历史版本不可变

- 已成功创建的文件版本不得覆盖、原地修改或复用版本号。
- 内容未变化时不得创建新版本；以 SHA-256 内容哈希判定。
- 改名、移动、删除和恢复都必须形成可审计事件。
- 本地删除只创建 tombstone，不删除历史 Blob。
- 历史版本永不自动清理。
- 允许用户手动永久删除，但必须显式确认、重新验证界面密码锁，并写入审计日志。

### 2.5 备份成功与预览成功相互独立

- 原始文件上传并校验成功后，版本即可进入 `backed_up` 状态。
- 预览、Office 转换、语法高亮或 Diff 失败不得回滚备份。
- 预览失败必须记录可重试状态和错误原因。
- Worker 不得修改原始 Blob。

### 2.6 界面密码锁不是文件加密

- 第一版只实现应用内的文件夹/文件界面访问锁。
- 服务端磁盘上的原始文件仍为明文。
- 文档、代码和 UI 中必须明确说明：该功能不能阻止 NAS 管理员、数据库管理员或拥有磁盘权限的人读取文件。
- 禁止把它描述为“端到端加密”“零知识加密”或“磁盘加密”。
- 不得在未单独设计密钥管理、迁移和恢复方案前加入透明加密。

### 2.7 关闭窗口不停止备份

- 关闭桌面窗口时进入系统托盘。
- Python Agent 继续监听、扫描、上传和重试。
- 只有用户执行“退出 DocVault”时，才停止 Agent 和桌面进程。

### 2.8 恢复必须安全

恢复到原路径时必须按顺序执行：

1. 下载并校验目标历史版本。
2. 检查目标路径当前状态。
3. 若当前文件存在且内容不同，先将当前内容备份成新版本。
4. 写入同目录临时文件。
5. 校验临时文件哈希。
6. 使用平台支持的原子替换方式替换目标文件。
7. 记录恢复来源版本、结果和错误。

禁止直接覆盖当前文件。恢复到可选路径时，必须由用户明确选择目标并确认覆盖策略。

## 3. 仓库边界

目标结构：

```text
docvault/
├── AGENTS.md
├── README.md
├── apps/
│   ├── client/
│   │   ├── README.md
│   │   ├── desktop/          # Tauri 2 + React + TypeScript
│   │   └── agent/            # Python 3.12+ 本地备份 Agent
│   └── server/               # FastAPI + PostgreSQL + Worker
├── packages/
│   ├── api-client/           # 根据 OpenAPI 生成，禁止手改生成文件
│   └── shared-schemas/       # 仅放协议级 Schema/枚举
├── infra/
│   ├── docker-compose.yml
│   ├── caddy/
│   └── libreoffice/
└── docs/
    ├── architecture.md
    ├── backup-semantics.md
    ├── security.md
    └── api.md
```

### 3.1 客户端职责

`apps/client/desktop` 负责：

- 原生窗口、系统托盘、开机启动和桌面通知。
- 文件库、历史、Diff、预览、任务中心和设置界面。
- 文件夹选择器和恢复目标选择器。
- 启动、监控并关闭 Python Agent。
- 界面密码锁交互。

`apps/client/agent` 负责：

- 跨平台文件监听。
- 10 分钟防抖和 30 分钟补偿扫描。
- 排除规则、文件类型与大小规则。
- SHA-256 哈希。
- SQLite 本地状态、任务队列和断网重试。
- 分块上传、下载和安全恢复。
- 只监听 `127.0.0.1` 的本地 API。

### 3.2 服务端职责

`apps/server` 负责：

- 单用户下的多设备注册与设备令牌认证。
- 文件树、版本、tombstone、Blob 引用和审计日志。
- 内容寻址存储和重复内容去重。
- 文件上传、下载、历史查询和永久删除。
- 文本 Diff 数据生成。
- Markdown、文本、代码、JSON、YAML、CSV、图片和 PDF 预览。
- DOCX、XLSX、PPTX 通过 LibreOffice headless 转换后预览。
- 异步预览任务及失败重试。

### 3.3 禁止共享的实现

客户端和服务端不得共享：

- SQLAlchemy 数据库模型。
- 客户端 SQLite 表定义。
- 服务端 PostgreSQL 表定义。
- 本地绝对路径处理代码。
- NAS 存储实现。
- 进程生命周期配置。

可以共享：

- API 路径和请求/响应 Schema。
- 错误代码。
- 上传协议。
- 任务、版本和预览状态枚举。
- OpenAPI 生成的 TypeScript 客户端。

## 4. API 与本地通信规则

### 4.1 桌面 UI 与本地 Agent

- Agent 仅监听 `127.0.0.1` 随机端口。
- 每次启动生成临时访问 Token。
- CORS 只允许当前桌面 UI 的明确来源。
- 不得绑定 `0.0.0.0`。
- 不得把临时 Token 写入日志、崩溃报告或长期配置。

### 4.2 客户端与 NAS 服务端

- 局域网内也使用 HTTPS。
- 设备使用可撤销的访问令牌；令牌不得明文写入普通日志。
- API 必须进行路径规范化、上传大小限制、MIME/扩展名校验和权限校验。
- 服务端不得信任客户端提交的绝对路径。
- API 变更先修改服务端 Schema，再重新生成客户端；禁止直接编辑生成代码。
- 破坏性 API 变更必须版本化或提供迁移期。

## 5. 数据与存储规则

### 5.1 内容寻址存储

- 原始内容以 SHA-256 为键存入 NAS 挂载目录。
- 推荐布局：`blobs/ab/cd/<full_sha256>`。
- 写入过程使用临时文件，完成校验后原子移动到最终位置。
- 相同哈希只保存一份 Blob，版本表通过引用关联。
- 数据库事务失败时不得留下“已成功版本但 Blob 不存在”的状态。
- Blob 垃圾回收只能由显式永久删除流程触发，第一版不得后台自动清理。

### 5.2 客户端 SQLite

至少保存：

- 设备和服务端配置。
- 备份根目录与排除规则。
- 已扫描文件状态。
- 防抖截止时间。
- 最近成功备份哈希。
- 上传、删除和恢复任务。
- 重试次数、下次重试时间和最后错误。

本地数据库迁移必须向前兼容，升级失败不得删除待上传任务。

### 5.3 服务端 PostgreSQL

至少建模：

- Device
- BackupRoot
- FileEntry
- FileVersion
- Blob
- PreviewArtifact / PreviewJob
- PasswordLockRule
- AuditEvent
- DeviceToken

所有服务端记录使用稳定 ID。时间统一保存为 UTC，展示时再转换为本地时区。

## 6. 第一版格式范围

### 6.1 备份、历史、预览与 Diff

- `.md`, `.txt`, `.json`, `.yaml`, `.yml`, `.csv`, `.xml`
- `.html`, `.css`, `.js`, `.ts`, `.py`, `.java`, `.go`, `.rs`
- `.toml`, `.ini`, `.log`

Diff 至少支持：

- Unified Diff。
- 左右并排 Diff。
- 忽略空白字符。
- 忽略行尾差异。
- 任意两个历史版本比较。
- 超大文本降级为下载，不得阻塞 Worker。

### 6.2 备份与预览，不要求 Diff

- `.pdf`
- `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`
- `.docx`, `.xlsx`, `.pptx`

Office 预览是第一版必须能力，使用服务端 LibreOffice headless 转换。转换结果是派生物，不得替代原始文件。

### 6.3 第一版不承诺

- 音频、视频预览。
- 压缩包内容浏览。
- 旧版二进制 Office 格式 `.doc`, `.xls`, `.ppt` 的稳定预览。
- 全文搜索、OCR、在线编辑、协作或跨设备同步。

## 7. 安全规则

- 不需要公网访问；默认部署目标是 NAS 或局域网服务器。
- 仍必须使用 HTTPS、设备认证和最小权限文件系统账户。
- NAS 数据目录、PostgreSQL 数据目录和配置目录分开挂载。
- 防止路径穿越、符号链接逃逸和大小写路径碰撞。
- 恢复时禁止写出用户授权的备份根目录，除非用户显式选择可选路径。
- 日志不得包含文件正文、密码、令牌、Cookie 或完整私密路径；调试日志也要脱敏。
- LibreOffice 转换进程必须设置超时、文件大小限制和独立临时目录。
- 对永久删除、恢复、密码锁修改、设备令牌操作写审计日志。

## 8. Agent 工作方式

### 8.1 开始编码前

代理必须先：

1. 阅读本文件、根 README 和目标子项目 README。
2. 用自己的话重述当前任务边界和涉及的不变量。
3. 列出计划修改的文件。
4. 给出可验证的验收标准。
5. 标出数据库迁移、API 兼容性、跨平台或数据丢失风险。

未经用户明确要求，不要一次实现多个里程碑。

### 8.2 修改原则

- 做满足验收标准的最小改动。
- 不顺便重构无关代码。
- 不更换技术栈或核心依赖，除非先解释影响并得到确认。
- 不用 mock 掩盖尚未实现的核心备份语义。
- 不吞掉异常；错误必须进入结构化日志和可观察状态。
- 任何可能丢失数据的代码路径必须有测试。
- 生成代码与手写代码分目录；生成文件禁止手改。

### 8.3 遇到错误时

代理应：

1. 解释根因。
2. 给出至少两种方案及影响。
3. 选择改动范围最小、数据风险最低的方案。
4. 先补充可复现测试，再修复。
5. 修复后运行相关测试、类型检查和构建。

不要使用“修复所有问题”式的大范围修改。

### 8.4 完成任务时

必须报告：

- 修改了哪些文件。
- 实现了什么，为什么这样实现。
- 运行了哪些检查及结果。
- 未解决风险和下一步。
- 是否改变了 API、数据库 Schema、存储格式或用户数据。

## 9. 测试要求

### 9.1 必须覆盖的行为

客户端 Agent：

- 连续文件事件只产生一个 10 分钟防抖任务。
- 进程重启后防抖和上传队列可恢复。
- 内容哈希相同不上传新版本。
- 断网、超时和服务端 5xx 使用有上限的指数退避。
- 文件在哈希或上传期间再次变化时不会提交错误内容。
- 30 分钟扫描能够发现漏掉的创建、修改、移动和删除。
- 恢复前自动备份当前内容，校验后原子替换。

服务端：

- 相同 Blob 去重，但版本记录独立。
- 不同设备的同路径文件互不关联。
- tombstone 不删除历史版本。
- 版本不可变。
- 预览失败不影响备份状态。
- Office 转换超时、崩溃或格式错误可被隔离和重试。
- 永久删除只删除无引用 Blob，并记录审计事件。
- 路径穿越和未授权设备请求被拒绝。

桌面 UI：

- 关闭窗口进入托盘而不是退出。
- 设备层级在文件树中不可省略。
- 密码锁文案不宣称文件已加密。
- 预览失败时仍可下载原文件和查看历史。
- 危险操作需要明确确认。

### 9.2 检查门槛

仓库初始化后应提供统一入口，例如：

```bash
make lint
make typecheck
make test
make build
```

若尚未建立统一命令，代理必须运行目标子项目现有的等价命令，并在结果中逐项列出。不得声称运行了未实际执行的检查。

## 10. Definition of Done

一个功能只有在满足以下条件时才算完成：

- 验收标准有自动化测试或明确的可重复人工验证步骤。
- 类型检查、静态检查、测试和相关构建通过。
- 数据失败路径和重试行为已处理。
- 没有破坏第 2 节的不变量。
- API、Schema 或配置变更已更新文档和示例。
- 跨平台差异已说明；至少在当前开发平台完成验证。
- 用户可见错误信息清晰，不泄露敏感信息。
- 没有把预览成功误当作备份成功的必要条件。

## 11. 明确禁止

- 自动把服务端版本同步到其他设备。
- 合并不同设备的同路径文件。
- 覆盖或重写历史版本。
- 自动清理永久历史。
- 将界面密码锁宣传为加密。
- 因预览失败删除或回滚原始备份。
- 在没有备份当前内容时覆盖原路径。
- 本地 Agent 监听局域网地址。
- 客户端和服务端共用数据库模型或本地路径逻辑。
- 为了让测试通过而删除数据完整性检查。
- 在未征得确认时扩大需求、添加同步、账号体系、公网部署、在线编辑或端到端加密。
